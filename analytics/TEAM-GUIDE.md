# دليل فريق تحليل بيانات الموقع

هالدليل لأعضاء الفريق الجداد اللي بدهم يحللوا بيانات استخدام موقع الدلائل الدراسية ويتدربوا على PostHog ويساهموا بالمشروع.

ما في حاجة تكون خبير من أول يوم. اقرأ القسم يلي بتحتاجه وارجع للباقي لاحقاً.

---

## 1) شو هدفنا

نحوّل سلوك الطلاب على الموقع إلى أرقام وأسئلة واضحة مثل:

- أي مواد ومحاضرات أكتر استخدام
- قديش الطلاب بيقعدوا فعلياً على المحتوى
- وين بيضيعوا أو بيتوقفوا (سكرول / وقت تركيز)
- كيف أداء الأسئلة والاختبارات الذاتية
- هل البحث عم يلاقي نتائج ولا عم يرجع فاضي
- وين في مشاكل تحميل أو تجربة سيئة

المخرج النهائي مو بس «شوفنا الداشبورد». المخرج تقرير تحليل + توصيات قابلة للتنفيذ للموقع.

---

## 2) الأدوات

### PostHog (الأساسي للتحليل)

برنامج تحليل منتج. كل حدث (event) إلو اسم وخصائص (properties) تقدر تسأل عنها مع بعض، مثلاً:

«متوسط وقت التركيز على `par9` في مادة معينة»

رابط المشروع: [eu.posthog.com](https://eu.posthog.com) (منطقة EU)

### Microsoft Clarity (مكمل)

تسجيلات جلسات وheatmaps. مفيد تشوف «كيف» الطالب بيتحرك على الصفحة، بس مو بديل عن PostHog للأسئلة الرقمية المعقدة.

## 4) كيف تبدأ بـ PostHog (بدون كود)

1. اطلب دعوة للمشروع من مسؤول الفريق
2. افتح **Activity → Live events** وتأكد إنو في أحداث واصمة
3. افتح أي event واقرأ الـ **properties** (مثل `subject`, `content_id`, `active_seconds`)
4. روح على **Product analytics / Insights** وجرب Trend بسيط على `content_viewed`
5. افتح **SQL** (HogQL) والصق واحد من ملفات `analytics/queries/`
6. Clarity اختياري: شوف تسجيلات لصفحات فيها سلوك غريب اكتشفتو بالأرقام

### مفاهيم لازم تفهمها

- **Event**: حدث حصل (مثلاً `mcq_answered`)
- **Properties**: تفاصيل الحدث (مثلاً `is_correct`, `subject`)
- **Dashboard**: مجموعة رسوم/استعلامات محفوظة
- **HogQL**: SQL داخل PostHog على جدول `events`

### أمثلة أسئلة تحليلوها أول أسبوع

1. أكثر 10 محاضرات حسب عدد الجلسات ووقت التركيز  
   → `analytics/queries/top-lectures.hogql`
2. مقارنة lectures vs DAWRAT vs notes  
   → `dawrat-vs-lectures.hogql`
3. دقة إجابات MCQ حسب المادة والمصدر  
   → `mcq-accuracy.hogql`
4. نسبة البحث بدون نتائج  
   → `search-findability.hogql`
5. نتائج الاختبارات الذاتية  
   → `exam-outcomes.hogql`

## 5) قاموس الأحداث (مختصر)

### تفاعل ودراسة

| الحدث | المعنى |
|--------|--------|
| `content_viewed` | فتح محتوى (رئيسية / محاضرة / دورات / ملاحظات) |
| `study_session_end` | نهاية جلسة دراسة + `active_seconds`, `max_scroll_pct`, `exit_scroll_pct` (وين وصل فعلياً لما طلع), `exit_reason` (`navigated_away`/`tab_closed`/`idle_timeout`), `engagement_level` (`studier`/`watcher`) |
| `scroll_milestone` | وصل 25/50/75/100٪ سكرول |
| `focus_milestone` | وصل دقيقة / 5 / 15 تركيز |

**studier vs watcher**: كل جلسة بتتصنف تلقائياً — `studier` إذا قعد ٣٠ ثانية+ وسكرول ٤٠٪+ (أو تفاعل فعلي زي إجابة سؤال / بحث / TOC)، غير هيك `watcher`. منيح لتمييز اللي فعلاً بيذاكر عن اللي بس فاتح الصفحة.

**زوار عائدون**: كل event فيه كمان `is_returning_visitor`, `visit_count`, `days_since_first_seen` (محسوبة من `localStorage`، بدون أي معلومة شخصية).

### تعلم وتجربة استخدام

| الحدث | المعنى |
|--------|--------|
| `mcq_answered` | جاوب سؤال (`is_correct`, `source`) |
| `exam_started` / `exam_finished` | بدأ/أنهى اختبار ذاتي |
| `lecture_progress_toggled` | علّم محاضرة مكتملة أو لا |
| `search_performed` | بحث (طول الاستعلام + عدد النتائج، بدون نص البحث الخام) |
| `search_result_clicked` | ضغط على نتيجة بحث |
| `toc_navigated` | تنقل من فهرس المحاضرة |
| `content_load_failed` | فشل تحميل |

خصائص مشتركة مهمة: `subject`, `storage_prefix`, `page`, `content_type`, `content_id`, `site_env`.

التفاصيل الكاملة بـ [`README.md`](README.md).

## 6) كيف تضيف ميزة تتبع جديدة (للمساهمين بالكود)

إذا اكتشفتوا سلوك مهم مو متعقب بعد:

1. اتفقوا على اسم الحدث والخصائص (بالعربي للمعنى، وبالإنجليزي للاسم التقني)
2. أضيفوا دالة بـ `site-shell/js/analytics.js` أو استخدموا `trackUxEvent(...)`
3. نادوها من المكان المناسب (`app.js` / `exam.js` / interactivity)
4. مرّروا دائماً السياق الكامل (`subject`, `content_type`…) مش بس رقم واحد
5. حدّثوا `analytics/README.md` + أضيفوا query بـ `analytics/queries/` إذا السؤال متكرر
6. اختبروا محلياً بـ `POSTHOG_KEY=...` وشوفوا Live events
7. افتحوا PR واضح: ليش الحدث + مثال سؤال التحليل

قواعد مهمة:

- لا ترسلوا معلومات شخصية (أسماء، إيميلات، نص بحث حرفي طويل)
- لا تعتمدوا على autocapture؛ نحن نرسل أحداث صريحة فقط
- Clarity بيستقبل نسخة مبسطة من نفس الإشارات؛ PostHog هو مصدر التحليل الرقمي

## 7) المخرج النهائي المطلوب من كل عضو / مجموعة

بنهاية فترة التدريب أو المهمة، لازم يطلع واحد من التالي (أو مزيج):

### أ) تقرير تحليل (مطلوب أساسي)

ملف Markdown أو PDF قصير (٢–٥ صفحات) فيه:

1. **سؤال البحث**: شو بدنا نعرف
2. **الفترة والفلاتر**: من–إلى، production فقط، مادة معيّنة…
3. **المنهج**: أي events / queries استخدمنا
4. **النتائج**: جداول أو لقطات شاشة من PostHog + أرقام واضحة
5. **التفسير**: شو يعني هالرقم للموقع وللطلاب
6. **توصيات**: ٣ اقتراحات عملية (محتوى، UX، تتبع ناقص…)
7. **حدود التحليل**: شو ما قدرنا نعرفه ولماذا
(هاد ما ضروري)
### ب) أصول قابلة لإعادة الاستخدام (مستحسن)

- ملف `.hogql` جديد ضمن `analytics/queries/`
- أو Insight/Dashboard محفوظ بـ PostHog مع اسم واضح
- أو PR صغير يضيف حدث تتبع ناقص اكتشفتوه بالتقرير

### ج) عرض شفهي قصير (اختياري)

٥–١٠ دقايق: السؤال → النتيجة → التوصية

## 8) معايير جودة التقرير

تقرير قوي:

- السؤال محدد مش عام («كل شيء»)
- الأرقام مربوطة بـ event properties مو انطباع شخصي
- في مقارنة (مادة vs مادة، أو قبل/بعد، أو نوع محتوى vs نوع)
- التوصية قابلة للتنفيذ خلال أسابيع

تقرير ضعيف:

- بس سكرين شوت بدون تفسير
- خلط sandbox/local مع production بدون تنبيه
- استنتاجات عن أشخاص بعينهم بدل أنماط عامة

فلتر مهم دائماً: `site_env = production` لما تحللوا استخدام حقيقي.
